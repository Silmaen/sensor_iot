# ESP32-C3 + BME280 + BH1750 + 2S Battery + Deep Sleep

ESP32-C3 Super Mini with BME280 (temperature, humidity, pressure), BH1750 (calibrated lux),
2S battery pack with ultra-low quiescent power chain, and deep sleep. See
[components inventory](../components.md) and [architecture](../architecture.md).

## Overview

| Parameter        | Value                                                   |
|------------------|---------------------------------------------------------|
| MCU              | ESP32-C3 Super Mini (RISC-V, WiFi+BLE, 3.3V native)     |
| Sensors          | BME280 (temp/humi/press) + BH1750 (lux), shared I2C bus |
| Power            | 2S 18650 → BMS 2S → HT7333 LDO (3.3V, 4µA Iq)           |
| Battery monitor  | Voltage divider + MOSFET switch on GPIO3/GPIO5          |
| Sleep mode       | ESP32 deep sleep (~5µA), timer wake                     |
| Publish interval | 300s (5 min), configurable via MQTT `set_interval`      |
| PlatformIO env   | `sensor_c3_bme280_bh1750`                               |
| HW code / rev    | `C3BMELUX` / rev 1                                      |
| Device ID        | Provisioned at runtime (no unit deployed yet)           |

## Wiring

![ESP32-C3 Configuration Wiring](../img/config-esp32c3.svg)

### GPIO Assignment

| GPIO  | Function      | Connected to           | Notes                          |
|-------|---------------|------------------------|--------------------------------|
| GPIO6 | I2C SDA       | BME280 SDA, BH1750 SDA | Shared I2C data bus            |
| GPIO7 | I2C SCL       | BME280 SCL, BH1750 SCL | Shared I2C clock bus           |
| GPIO3 | ADC (battery) | Voltage divider mid    | ADC1_CH3, 12-bit, no strapping |
| GPIO5 | MOSFET gate   | Q1 gate (2N7000)       | Divider switch control         |
| 3V3   | Power         | HT7333 OUT             | 3.3V regulated supply          |
| GND   | Ground        | Common ground          |                                |

> **Strapping pins to avoid:** GPIO0, GPIO2 (boot mode), GPIO8 (pull-up), GPIO9 (boot button).

### I2C Bus

Both sensors share the I2C bus — no address conflict:

| Device | I2C Address | Pull-ups     |
|--------|-------------|--------------|
| BME280 | 0x76        | 4.7k to 3.3V |
| BH1750 | 0x23        | (shared)     |

> **Pull-ups:** Add 4.7k resistors on SDA and SCL to 3.3V if not already present on breakout boards.
> Most BME280 breakouts include them; verify BH1750 breakout.

### Power Chain

```text
2x 18650 (series, 6.0-8.4V)
    │
    ├── BMS 2S (protection: overcharge/overdischarge/short)
    │       │
    │       ├── P+ ──┬── HT7333 IN → 3.3V OUT → ESP32-C3 3V3 + sensors
    │       │        │
    │       │        └── R1 (22k) → MID → R2 (12k) → Q1 (2N7000) → GND
    │       │                         │                    │
    │       │                    GPIO3 (ADC)          GPIO5 (gate)
    │       │
    │       └── P- ── GND (common)
```

**Key advantage over ESP8266 configs:** No 5V regulator needed. The ESP32-C3 runs natively at
3.3V, and both sensors are 3.3V. The HT7333 LDO has only ~4 µA quiescent current.

### Battery Voltage Divider

Same divider as ESP8266 configs (22k/12k), but read by the ESP32-C3 12-bit ADC (0-4095 vs 0-1023).

| Parameter       | Value                                                      |
|-----------------|------------------------------------------------------------|
| R1              | 22 k-ohm (top, to V_bat)                                   |
| R2              | 12 k-ohm (bottom, to Q1 drain)                             |
| Divider ratio   | 2.833 nominal (runtime `bat_divider`, calibrated per unit) |
| ADC resolution  | 12-bit (0-4095)                                            |
| ADC ref voltage | 3.3V                                                       |
| V_adc at 8.4V   | 2.96V (ADC ~3676)                                          |
| V_adc at 6.0V   | 2.12V (ADC ~2630)                                          |
| MOSFET switch   | 2N7000 on GPIO5 (0µA in sleep)                             |

## Deep Sleep

The ESP32-C3 deep sleep works like ESP8266 — full reboot on wake. RTC memory (`RTC_DATA_ATTR`)
persists the publish interval, battery calibration ratio, and the WiFi BSSID cache
(see [WiFi & Connectivity](#wifi--connectivity)) across sleep cycles.

| State               | Current draw | Duration |
|---------------------|--------------|----------|
| Deep sleep          | **~9 µA**    | 295.5s   |
| WiFi connect        | ~80 mA       | 1-3s     |
| MQTT + sensors      | ~80 mA       | 1.5s     |
| **Average (5 min)** | **~1.21 mA** |          |

### Autonomy Estimate (2S 2200 mAh)

| Phase                    | Charge per cycle |
|--------------------------|------------------|
| Active (4.5s × 80mA)     | 360 mA·s         |
| Sleep (295.5s × 0.009mA) | 2.7 mA·s         |
| **Total per 300s cycle** | **363 mA·s**     |
| **Average current**      | **1.21 mA**      |
| **Autonomy**             | **~76 days**     |

Compare with ESP8266 HW rev 1 + MC78M05BTG: ~21 days theoretical / ~20 days measured. This config
is **~3.6× better** — it already has the low-Iq regulator (HT7333) and MOSFET-switched divider that
the ESP8266 nodes only get at HW rev 2. **Theoretical only**: no C3 unit is deployed yet, so there
is no field-measured figure.

## PlatformIO Configuration

```ini
[env:sensor_c3_bme280_bh1750]
extends = common_esp32c3
build_flags =
    ${common_esp32c3.build_flags}
    -DMQTT_DEVICE_TYPE='"thermo"'
    -DHW_CODE='"C3BMELUX"'
    -DHW_REV=1
    -DHAS_BME280
    -DHAS_BH1750
    -DHAS_BATTERY
    -DHAS_DEEP_SLEEP
    -DHAS_CALIBRATION
    -DHAS_OTA
```

`device_id` and the battery divider ratio are **not** compiled in — they live in the runtime
config store (NVS on the C3) and survive OTA. Provision identity once over serial
(`provision <id>`), then calibrate the divider with `calibrate_battery`. See the
[OTA module](../modules/ota.md) and [calibration](../modules/calibration.md).

> **Note (HW rev 1):** the C3 uses a 12-bit ADC. A per-unit `bat_divider` calibration is required
> at commissioning — an uncalibrated node over-reads battery voltage. Run `calibrate_battery` with a
> known pack voltage after provisioning.

Build and upload:

```bash
pio run -e sensor_c3_bme280_bh1750              # Build
pio run -e sensor_c3_bme280_bh1750 -t upload    # Upload via USB-C
pio device monitor                               # Serial monitor (115200)
```

## WiFi & Connectivity

The C3 joins the same 2.4 GHz WiFi as the ESP8266/MKR nodes, but the ESP32 Arduino core is
stricter in two ways the ESP8266 core handles transparently — both must be set explicitly:

1. **AP channel for a hidden SSID.** The ESP32 discovers an AP by scanning, but a hidden SSID
   broadcasts no name, so `WiFi.begin(ssid, pass)` fails with `WL_NO_SSID_AVAIL` (reason 201).
   Passing the channel makes it associate directly. → Set `WIFI_CHANNEL` in `credentials.h`
   (`0` = auto; set it to the AP's channel for a hidden SSID).
2. **Lower minimum security for WPA-PSK (WPA1).** The ESP32 default minimum is WPA2-PSK, so a WPA1
   AP is silently filtered out (also reason 201). → The driver calls
   `WiFi.setMinSecurity(WIFI_AUTH_WPA_PSK)` in `esp32_network.cpp`.

> This is why the 8266 nodes join a hidden/WPA1 network out of the box while the C3 needs both.

### Roaming with an RTC BSSID cache

The network may have several APs broadcasting the same (hidden) SSID on one channel. To always
associate with the **strongest** AP, without pinning a single BSSID (the node may be moved, slowly):

- **Full scan** (`WIFI_ALL_CHANNEL_SCAN` + `WIFI_CONNECT_AP_BY_SIGNAL`) on cold boot, every 12th
  connect, or when a cached connect fails → picks the strongest AP and caches its **BSSID + channel
  in RTC memory**.
- **Fast path** on the intermediate deep-sleep wake-ups: reconnect straight to the cached BSSID
  (~0.2 s vs ~2.5 s for a full scan). The full scan's extra radio-on time costs ~35 % of autonomy —
  hence the cache.
- A failed cached connect falls back to a full rescan, so roaming still happens (correctness-safe).

Implemented in `esp32_network.cpp::connect_wifi()`.

## Antenna (WiFi range)

The ESP32-C3 Super Mini's on-board ceramic antenna is poor: right next to an AP it reads only
~-70 dBm and the uplink is erratic (marginal association, packet loss). For anything beyond a few
metres — especially outdoors — replace it with a quarter-wave wire.

**Mod:** solder a **31 mm** wire (¼λ at 2.4 GHz) to the antenna **feed pad** — the pad on the
white-marked end of the ceramic antenna (near GPIO0); the opposite end is a mount/ground point.
Leave the ceramic in place (it becomes electrically inert) or remove it. Keep the wire straight,
vertical, and clear of metal (battery pack, enclosure). Reference:
[peterneufeld.wordpress.com](https://peterneufeld.wordpress.com/2025/03/04/esp32-c3-supermini-antenna-modification/)
(1.0 mm silver loop); a plain 0.4 mm straight monopole (e.g. a Cat6 conductor) also works well.

**Measured (same spot, hidden AP):**

| Antenna       | RX scan | RX connected | Uplink (round-trip) |
|---------------|---------|--------------|---------------------|
| Stock ceramic | -66 dBm | -65 dBm      | 3/10 — lossy        |
| ¼λ wire mod   | -48 dBm | -53 dBm      | 10/10 — healthy     |

→ ~+18 dB RX and a reliable uplink.

> ⚠️ **Validate an antenna mod by actually CONNECTING, not just scanning.** A scan only exercises RX;
> a cold or badly-fed joint gives great RX but dead TX — association then fails (reason 201/39). A
> weak solder, the wire on the wrong pad, or a wire shorted to ground are the usual culprits.

### Diagnostics & bring-up

- **`wifi_scan_c3`** env — scans every network (RSSI / channel / BSSID, incl. hidden), then connects
  to the target SSID and runs a broker round-trip test, reporting RX (dBm) and an uplink-health proxy
  (TX dBm is not observable from the STA). Use it to measure both link directions after an antenna mod.
- **Verbose serial:** temporarily comment `-DHAS_DEEP_SLEEP` and add `-DHAS_SERIAL_DEBUG` in the env
  (stays awake so the native USB-CDC link is not dropped each sleep cycle; 10 s interval).

> **USB notes (Super Mini):** the board exposes the C3's *native* USB. PlatformIO cannot auto-detect
> it — pass `--upload-port /dev/ttyACM0`. A **USB-A → USB-C** cable is required: a C-to-C cable will
> not enumerate (the board omits the CC pull-down resistors). The USB-C connector is mechanically
> fragile, but this only affects bench debugging — a deployed node runs on battery + WiFi, no USB.

## Troubleshooting

| Symptom                                  | Cause                                    | Fix                                                                                                                  |
|------------------------------------------|------------------------------------------|----------------------------------------------------------------------------------------------------------------------|
| Board not detected on USB                | C-to-C cable, or port auto-detect        | Use a USB-A→C cable (C-to-C won't enumerate); pass `--upload-port /dev/ttyACM0`                                      |
| Upload fails                             | Board not in download mode               | Hold BOOT while pressing RST, then upload                                                                            |
| WiFi fails, `reason=201` (NO_AP_FOUND)   | Hidden SSID and/or WPA1 not handled      | Set `WIFI_CHANNEL` for a hidden AP; firmware lowers min-security to WPA_PSK. Verify SSID/channel with `wifi_scan_c3` |
| Strong RSSI but won't associate (201/39) | Antenna mod: cold joint / dead TX        | Reflow the feed joint; ensure the wire is on the feed pad and not shorted to GND                                     |
| Connects but lossy / drops               | Weak antenna (stock ceramic)             | See [Antenna](#antenna-wifi-range); aim for RSSI > -70 dBm                                                           |
| BME280 init failed                       | I2C address or wiring                    | Verify 0x76 (SDO→GND), check SDA/SCL on GPIO6/7                                                                      |
| BH1750 init failed                       | ADDR pin floating                        | Connect ADDR to GND for 0x23                                                                                         |
| Battery reads 0                          | MOSFET not wired or GPIO5                | Verify 2N7000 gate→GPIO5, drain→R2, source→GND                                                                       |
| Very short battery life                  | Wrong regulator (78M05)                  | Use HT7333 (4µA Iq), NOT MC78M05BTG (3mA Iq)                                                                         |
| `batv` too high / ADC inaccurate         | ESP32-C3 ADC non-linearity, uncalibrated | Calibrate with `calibrate_battery` (a fresh node over-reads)                                                         |
